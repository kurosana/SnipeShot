変更前後で毎回このケースを実行し、期待値と一致することを確認する。  
一騎打ち・ドボンの条件ロジックを変更した場合は、このファイルに変更内容と新しい期待値を追記すること。

## 一騎打ち（`FilterEngine.shouldDuel` / `countResults`）

### 判定ルール

- **一騎打ち発動**: `lines === 2`（`shouldDuel`）
- **ライン数 `lines`**: `classifyHits` による
  - **単独ライン**: ヒット集合内で `e` が1要素のポケモンが持つラインIDの集合（`singleton.size`）
  - **残り分岐**: ヒット内に単独ラインの葉が1つもない複数所属ポケモン（`remaining`）を、共有ラインで連結成分に分割した数
  - `lines = singleton.size + multiComponents`
- **吸収**: 分岐前ポケモン（例: カラカラ）が、ヒット内の単独ライン葉（例: ガラガラ）と同じ行にまとまる。ライン数には単独ラインとして数えない

### 過去に引っかかった点

| 問題 | 原因 | 修正後の期待 |
|------|------|--------------|
| 進化ライン数が18など異常に少ない | `e`+技セットでグループ化していた | 全件で `lines=604`（mode_all.json） |
| カラカラのみで一騎打ちになる | 分岐前を複数ラインとして数えていた | `lines=1`, `duel=false` |
| カラカラ+両ガラガラで一騎打ちにならない | 吸収ロジック未実装 | `lines=2`, `duel=true` |
| 両ガラガラのみで一騎打ちにならない | 同上 | `lines=2`, `duel=true` |
| タマタマのみで2ラインになる | 分岐フォルムを別ライン扱い | `lines=1` |
| イーブイのみで8ラインになる | 全進化先ラインを重複カウント | `lines=1`（イーブイ単体ヒット時） |

## ドボン（`checkDobon` in `js/app.js`）

### 判定ルール

`refreshResults` 内で、次をすべて満たすとき `appendDobon()`:

1. `prevCount` / `prevLines` が null でない（初回絞り込み後）
2. `hasActiveConditions()` — 除外されていない完成済み条件が1つ以上
3. `conditionsChanged` — 条件シグネチャが前回と異なる
4. `lines === 0 || lines === 1`（一騎打ち未満に留まっている）
5. `count >= prevCount` **または** `prevCount !== count || prevLines !== lines`（匹数が減らない、または何らかの統計変化）

**ドボンを出さない場合**

- 初回の条件適用（`prevResultStats` 未設定）
- 条件が変わっていない再計算
- `lines >= 2`（一騎打ちラインに到達）
- 有効な条件が0件
- `refreshResults({ skipDobon: true })` — カンニング ON 時

### 過去に引っかかった点

| 問題 | 原因 | 修正後の期待 |
|------|------|--------------|
| 初回絞り込みでドボンが出ない | 旧ロジックが `count >= prevCount && lines <= 1` のみ | 条件変更かつ `lines` 0〜1 なら、匹数が減ってもドボン可 |
| 条件変更なしでドボンが出る | `conditionSignature` 未使用 | 条件変更時のみ |
| カンニングでドボンが出る | skipDobon 未対応 | カンニング時は出ない |
| 除外してもドボンが残る | clearDobonStack 未呼び出し | 除外ボタンでスタック消去 |
| 複数回ドボンできない | 単一フラグ | 条件変更のたびにスタック追加可 |

### ドボン ロジック単体テスト（シミュレーション）
